# Image Blur Editor UX 기획

## 1. 문서 목적

이 문서는 Image Blur Editor의 사용 흐름과 상호작용 원칙을 정의한다.

목표는 다음과 같다.

- 로그인 없이 바로 시작하는 로컬 편집 경험을 설계한다.
- 초보 사용자도 망설이지 않고 첫 결과를 만들 수 있게 한다.
- 도구의 자유도는 유지하되, 작업 흐름은 단순하게 만든다.

## 2. Dieter Rams 조사 기반 UX 해석

Rams는 산업 디자이너였지만, 그의 원칙은 UX에도 직접 적용할 수 있다.

이번 앱에서 UX로 번역되는 핵심 해석은 아래와 같다.

### 2-1. Useful → 모든 단계는 작업 효율을 높여야 한다

- 업로드
- 도구 선택
- 적용
- 되돌리기
- 다운로드

이 흐름이 가장 짧고 분명해야 한다.

### 2-2. Understandable → 사용법을 외우지 않아도 이해되어야 한다

- 첫 화면만 봐도 "업로드 후 편집하고 다운로드" 흐름이 읽혀야 한다
- 현재 선택 상태를 항상 보여줘야 한다

### 2-3. Unobtrusive → 인터페이스는 조용해야 한다

- 사용자가 앱을 배워야 하는 것이 아니라 이미지를 편집하면 된다
- 설명은 충분하되 과하게 개입하지 않는다

### 2-4. Honest → 효과의 용도와 한계를 숨기지 않는다

- `Blur / Smooth`, `Pixelate`는 미감용/일반 가림용
- 민감정보에는 `Blur / Solid Fill` 권장
- 보안성 한계를 UX 카피로 정직하게 전달한다

### 2-5. Long-lasting → 유행성 패턴보다 익숙하고 오래 가는 흐름을 택한다

- 편집기에서 기대되는 도구 구조를 존중한다
- `Undo/Redo`, `Brush`, `Rectangle`, `Download`처럼 보편적 패턴을 채택한다

### 2-6. As Little Design As Possible → 행동 수를 줄이고 본질만 남긴다

- 초기 진입 단계는 하나: 업로드
- 편집 단계는 셋: 선택, 적용, 저장
- 고급 옵션은 조건부 노출

## 3. UX 목표

이 앱의 핵심 UX 목표는 아래 세 가지다.

- 10초 안에 첫 편집을 시작할 수 있어야 한다
- 첫 편집 결과를 실수 없이 다운로드할 수 있어야 한다
- 여러 효과를 섞어 써도 정신적으로 복잡하지 않아야 한다

## 4. 핵심 사용자 흐름

### 4-1. 첫 진입 흐름

1. 사용자는 빈 편집 화면을 본다
2. 가장 눈에 띄는 행동은 `이미지 업로드`
3. 설명은 한 줄 또는 두 줄이면 충분하다
4. 업로드 직후 바로 편집 가능한 상태로 전환한다

UX 원칙:

- 빈 상태에서 선택지를 늘리지 않는다
- 시작 버튼은 하나면 충분하다

### 4-2. 기본 편집 흐름

1. 이미지 업로드
2. 도구 선택
3. 효과 선택
4. 강도/크기 조절
5. 클릭 또는 드래그로 적용
6. 필요 시 Undo/Redo
7. 다운로드

UX 원칙:

- 도구와 효과를 동시에 바꾸지 않아도 되게 한다
- 현재 상태를 항상 보이게 해 혼란을 줄인다

### 4-3. 문서/스크린샷 가림 흐름

1. `Rectangle` 선택
2. `Blur / Solid Fill` 또는 `Pixelate` 선택
3. 필요한 영역 드래그
4. 바로 다음 영역 처리 반복
5. 다운로드

이 흐름은 반복성이 중요하므로, 한 번 적용 후 다시 같은 도구 상태를 유지해야 한다.

### 4-4. 얼굴/배경 블러 흐름

1. `Brush` 선택
2. `Blur / Smooth` 선택
3. 브러시 크기 조절
4. 원하는 영역을 드래그로 칠함
5. 필요 시 여러 번 덧칠
6. 다운로드

이 흐름은 브러시 프리뷰와 드래그 안정성이 중요하다.

## 5. 상호작용 원칙

### 5-1. 직접 조작 중심

- 사용자는 캔버스를 직접 만지듯 편집해야 한다
- 버튼을 많이 누르는 경험보다 캔버스 상호작용이 중심이어야 한다

### 5-2. 현재 상태는 항상 명확해야 한다

항상 알 수 있어야 하는 정보:

- 현재 도구
- 현재 효과
- 현재 강도 또는 크기
- Undo 가능 여부
- 다운로드 가능 여부

추가 원칙:

- 클릭 가능한 요소는 한눈에 클릭 가능해 보여야 한다
- 버튼과 정보 요소가 시각적으로 혼동되면 사용 흐름이 느려진다
- 사용자는 먼저 "누를 수 있는 것"을 읽고, 그다음 "읽기만 하면 되는 것"을 읽어야 한다

### 5-3. 효과는 예측 가능해야 한다

- 클릭은 1회 적용
- 드래그는 연속 적용
- 사각형은 영역 지정 후 정확히 그 범위에 적용

사용자가 "왜 저렇게 됐지?"라고 느끼지 않게 만들어야 한다.

### 5-4. 작업은 쉽게 되돌릴 수 있어야 한다

- 클릭 1회 = Undo 1단위
- 드래그 1회 = Undo 1단위
- 사각형 1회 = Undo 1단위

시간 기반이 아니라 행동 기반이 자연스럽다.

## 6. 도구별 UX 규칙

### 6-1. Brush

- 브러시 커서 프리뷰를 제공한다
- 드래그 중에는 끊김이 적어야 한다
- 빠르게 여러 번 칠할 수 있어야 한다

필수 피드백:

- 커서 크기
- 현재 효과
- 적용 즉시 보이는 시각 변화

### 6-2. Rectangle

- 시작점과 종료점이 명확해야 한다
- 드래그 중 임시 윤곽선을 보여준다
- 적용 범위가 예측 가능해야 한다

문서형 작업에서 신뢰감이 중요하다.

## 7. 효과별 UX 규칙

### 7-1. Blur / Smooth

- 감성 편집 또는 얼굴 블러에 적합하다는 인식을 준다
- 민감정보 보호용으로 과장하지 않는다

### 7-2. Pixelate

- 전통적인 모자이크 도구로 인식되게 한다
- 텍스트나 UI 일부 가림에 잘 맞는다고 안내한다

### 7-3. Blur / Solid Fill

- 가장 확실한 가림 도구라는 메시지를 준다
- 민감정보 처리에는 기본 권장 도구로 위치시킨다

## 8. 상태 피드백 설계

### 8-1. 상태 라인

헤더 또는 패널 상단에 짧은 상태 텍스트를 둔다.

예시:

- `파일명 · 1920 x 1080`
- `이미지를 불러오는 중입니다`
- `마지막 편집을 되돌렸습니다`
- `다운로드 완료`

### 8-2. 과도한 알림 대신 조용한 피드백

- 토스트 남발 금지
- 모달 남발 금지
- 대부분의 상태는 인라인 텍스트로 충분하다

단, 조용하다는 이유로 클릭 요소의 존재감까지 약해지면 안 된다.

### 8-3. 실패 상황

실패 상황은 짧고 구체적으로 안내한다.

예시:

- 이미지 로드 실패
- 내보내기 실패
- 지원되지 않는 브라우저에서 `Blur / Smooth` 비활성

## 9. 온보딩 방식

로그인 없는 정적 웹앱이므로 온보딩도 매우 가벼워야 한다.

### 9-1. 첫 진입 도움말

- 업로드 버튼 근처 짧은 설명
- 캔버스 빈 상태에 한두 줄 가이드

### 9-2. 학습 부담 최소화

- 별도 튜토리얼 모달은 두지 않는다
- 기본 흐름 자체가 학습 역할을 해야 한다

### 9-3. 도움말 우선순위

- 먼저 행동을 유도한다
- 그 다음 단축키를 보여준다
- 마지막에 보안 안내를 보여준다

## 10. 보안성과 정직한 UX 카피

이 앱은 다양한 사용자 대상이므로, 가림 효과의 성격을 정확히 알려야 한다.

원칙:

- 블러는 "예뻐 보이는 숨김"일 수 있다
- 완전한 민감정보 보호는 `Blur / Solid Fill`이 더 적합하다
- 경고 문구는 공포 조장이 아니라 정직한 안내여야 한다

예시 카피 방향:

- `민감정보에는 Blur / Solid Fill을 권장합니다`
- `Blur / Smooth는 시각적 흐림 효과입니다`

## 11. 단축키 UX

단축키는 생산성을 높이되, 몰라도 앱 사용이 가능해야 한다.

기본 단축키:

- `B`: Brush
- `R`: Rectangle
- `1`: Blur / Smooth
- `2`: Pixelate
- `3`: Blur / Solid Fill
- `Ctrl+Z`: Undo
- `Ctrl+Shift+Z` 또는 `Ctrl+Y`: Redo
- `[` / `]`: 브러시 크기 조절

노출 방식:

- 패널 하단 또는 도움말 영역에 항상 짧게 노출
- 툴팁 없이도 알아볼 수 있는 수준으로 유지

## 12. 접근성과 편의성

### 12-1. 키보드 접근

- 주요 컨트롤은 키보드 포커스로 이동 가능해야 한다
- 단축키와 포커스 이동이 충돌하지 않게 설계한다

### 12-2. 대비

- 중성 색을 쓰더라도 텍스트 대비는 충분해야 한다
- 비활성 상태와 선택 상태를 명확히 구분해야 한다

### 12-3. 포인터 신뢰성

- 드래그는 끊기지 않아야 한다
- 포인터 캡처를 사용해 캔버스 밖으로 나가도 안정적이어야 한다

## 13. UX 안티패턴

- 첫 화면에서 옵션이 너무 많은 상태
- 현재 선택 도구가 보이지 않는 상태
- 드래그 결과가 예측 불가능한 상태
- Undo가 시간 기반처럼 들쑥날쑥한 상태
- 민감정보 보호 한계를 숨기는 카피
- 다운로드 결과와 화면 결과가 다른 상태

## 14. UX 구현 우선순위

### 14-1. 1차

- 업로드 → 편집 → 다운로드 흐름 단순화
- 드래그 안정성 확보
- Undo/Redo 일관성 확보
- 현재 상태 표시 강화

### 14-2. 2차

- 보안 안내 카피 정교화
- Empty/Loading/Error 상태 개선
- 단축키 노출 방식 정리

### 14-3. 3차

- 세부 마이크로카피 개선
- 고급 옵션의 접힘/펼침 방식 개선

## 16. 현재 결론

이 앱의 UX는 "기능이 많은 편집기"가 아니라 "바로 쓰이고 쉽게 되돌릴 수 있는 편집기"를 목표로 한다.

핵심은 다음과 같다.

- 시작은 빠르게
- 사용은 조용하게
- 상태는 분명하게
- 결과는 정직하게
- 고급스러움은 장식이 아니라 질서에서 나오게 한다
