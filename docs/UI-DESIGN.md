# Image Blur Editor UI 디자인 기획

## 1. 문서 목적

이 문서는 Image Blur Editor의 시각 구조와 화면 설계 원칙을 정의한다.

목표는 다음과 같다.

- 꾸밈보다 기능 전달이 앞서는 화면을 만든다.
- Dieter Rams의 원칙을 웹 기반 편집기에 맞게 번역한다.
- 데스크톱 전용 편집기의 정보 구조와 컴포넌트 기준을 정한다.

이 문서는 "예쁜 화면"보다 "오래 써도 피로하지 않고 기능이 잘 드러나는 화면"을 목표로 한다.

## 2. Dieter Rams 조사 요약

공식 자료를 기준으로 해석했을 때, 이번 앱에 직접 연결되는 핵심 원칙은 아래와 같다.

### 2-1. Useful

- 좋은 디자인은 제품의 유용성을 강화하고, 유용성을 해치는 요소를 덜어내야 한다.
- 따라서 이 앱의 UI는 블러/가림 작업을 방해하는 장식적 요소를 최소화해야 한다.

### 2-2. Understandable

- 좋은 디자인은 제품 구조를 명확히 보여주고, 사용법이 스스로 드러나야 한다.
- 따라서 툴, 효과, 강도, 상태, 다운로드 경로가 화면에서 자연스럽게 읽혀야 한다.

### 2-3. Unobtrusive

- Rams는 목적을 가진 제품은 도구처럼 조용해야 한다고 본다.
- 따라서 UI는 사용자의 이미지보다 먼저 튀면 안 된다.
- 앱은 무대가 아니라 도구여야 한다.

### 2-4. Honest

- 실제보다 더 좋아 보이게 과장하지 않아야 한다.
- 따라서 버튼, 뱃지, 경고, 효과 이름은 기능을 솔직하게 설명해야 한다.
- 예: `Blur / Smooth`는 미감용일 수 있으나 민감정보 완전 보호 수단처럼 보이게 표현하지 않는다.

### 2-5. Thorough Down To The Last Detail

- 정렬, 간격, 크기, 색, 상태 표현은 우연이 아니라 체계여야 한다.
- 따라서 UI는 spacing, stroke, radius, 상태 피드백을 하나의 시스템으로 묶어야 한다.

### 2-6. As Little Design As Possible

- 중요한 것은 적게 보이는 것이 아니라, 핵심만 남기는 것이다.
- 따라서 기능은 충분히 제공하되, 노출 방식은 절제되어야 한다.

## 3. 적용 해석

이번 앱에서 Dieter Rams 스타일은 아래처럼 해석한다.

- Braun 제품처럼 보이게 흉내 내는 것이 목표가 아니다.
- 웹앱에서의 Rams 해석은 `질서`, `비례`, `조용함`, `정직함`, `지속성`이다.
- 결과적으로 화면은 단순해야 하지만 빈약하면 안 된다.
- 단순함은 기능 삭제가 아니라 구조 정리에서 나온다.

## 4. UI 핵심 원칙

### 4-1. 이미지를 주인공으로 둔다

- 업로드된 이미지가 시선의 중심이 되어야 한다.
- 패널은 보조 도구로 존재하고, 캔버스를 압도하지 않는다.

### 4-2. 정보 위계를 분명히 한다

- 1차 정보: 현재 이미지, 현재 도구, 현재 효과, 다운로드
- 2차 정보: 강도, 크기, 색상
- 3차 정보: 도움말, 보안 안내, 단축키

### 4-3. 조용한 인터페이스를 유지한다

- 강한 포인트 컬러 남용 금지
- 과한 그림자, 유리 효과, 과장된 애니메이션 금지
- 상태 강조는 색보다 대비와 위치로 해결한다

### 4-4. 도구는 시스템처럼 보여야 한다

- 버튼, 슬라이더, 패널이 제각각 보이면 안 된다.
- 컨트롤은 명확한 규칙 속에서 반복되어야 한다.

### 4-5. 버튼은 버튼처럼 보여야 한다

- 클릭 가능한 요소는 텍스트나 정보 카드처럼 보이면 안 된다.
- 버튼은 배경, 테두리, 높이, 내부 여백, hover 상태로 즉시 식별 가능해야 한다.
- 특히 `Upload`, `Download`, `Undo`, `Redo`는 사용자가 한눈에 "누를 수 있는 컨트롤"로 인식해야 한다.
- Rams 스타일은 버튼을 숨기는 것이 아니라 과장 없이 명확하게 만드는 것이다.

## 5. 레이아웃 방향

데스크톱 전용 기준으로 4구역 구조를 채택한다.

### 5-1. 상단 헤더

- 좌측: 앱 이름 또는 워드마크
- 중앙 또는 좌측: 현재 파일명, 이미지 해상도
- 우측: 업로드, 다운로드, 초기화

헤더는 얇고 단정해야 하며, 제품의 정체성과 현재 작업 맥락만 전달한다.

### 5-2. 좌측 툴 레일

- `Brush`
- `Rectangle`
- `Undo`
- `Redo`

좌측에는 작업 행위 자체를 바꾸는 도구만 둔다.

이유:

- 사용자는 "무엇으로 편집할지"를 먼저 고른다.
- 도구와 속성을 분리하면 구조가 훨씬 이해하기 쉬워진다.

### 5-3. 중앙 캔버스 영역

- 가장 넓은 면적을 차지해야 한다.
- 이미지 주변 여백은 충분히 두되, 과도하게 비워 보이게 하지 않는다.
- 캔버스 배경은 이미지와 경계가 분명한 중성 톤을 사용한다.

### 5-4. 우측 속성 패널

- 효과 종류
- 블러 강도 또는 픽셀 크기
- 브러시 크기
- Redaction 색상/불투명도
- 도움말 및 보안 안내

우측에는 "선택된 도구/효과에 따라 달라지는 정보"만 둔다.

## 6. 정보 구조

### 6-1. 화면에서 항상 보여야 하는 것

- 현재 이미지 이름
- 현재 해상도
- 현재 도구
- 현재 효과
- Undo/Redo 가능 여부
- 다운로드 버튼

### 6-2. 조건부로 보여야 하는 것

- `Blur / Smooth` 선택 시 블러 강도
- `Pixelate` 선택 시 블록 크기
- `Blur / Solid Fill` 선택 시 색상과 불투명도
- `Brush` 선택 시 브러시 커서 프리뷰

### 6-3. 기본 접힘 또는 약한 강조로 충분한 것

- 단축키 설명
- 민감정보 보안 가이드
- 드래그/사각형 사용 팁

## 7. 시각 시스템

### 7-1. 색상 방향

전체 색상은 따뜻한 중성 계열을 기본으로 한다.

추천 방향:

- 배경: 웜 라이트 그레이 또는 아이보리
- 패널: 거의 흰색에 가까운 중성 톤
- 텍스트: 진한 흑색 대신 소프트한 차콜
- 강조: 아주 제한된 딥 블루 또는 다크 그레이
- 위험/경고: 채도 높은 빨강 대신 낮은 채도의 경고색

목표:

- 눈에 거슬리지 않고 오래 써도 피로가 적어야 한다.
- 화면이 이미지 편집 도구답게 조용해야 한다.

### 7-2. 색상 구조

색은 장식이 아니라 역할 체계로 정의한다.

#### Base

- 앱 전체 바탕과 큰 면적을 담당한다.
- 눈에 오래 노출되는 영역이므로 가장 조용해야 한다.
- 예: 전체 배경, 캔버스 주변 바닥면

#### Surface

- 실제 UI 블록의 덩어리를 구분한다.
- Base보다 밝고 안정적이어야 한다.
- 예: 헤더, 좌측 레일, 우측 패널, 상태 바

#### Control

- 눌릴 수 있는 요소를 구분한다.
- Surface 위에서 충분히 떠 보여야 하며, 정보 카드와 달라야 한다.
- 예: 버튼, 세그먼트, 툴 버튼

#### Accent

- 현재 선택 상태나 가장 중요한 액션을 강조한다.
- 남용하면 전체 위계가 흐려지므로 아주 제한적으로 사용한다.
- 예: Primary 버튼, 활성 도구, 선택된 효과

#### Feedback

- 성공, 경고, 비활성, 제한 상태를 설명한다.
- 강한 색보다 명확한 대비와 보조 텍스트를 우선한다.

### 7-3. 색상 적용 규칙

- Base와 Surface는 같은 계열의 중성 톤으로 묶고, 지나친 대비를 만들지 않는다.
- Control은 Surface보다 한 단계 더 선명하게 보여야 한다.
- Accent는 "현재 선택"과 "가장 중요한 행동"에만 쓴다.
- 안내 문구와 경고 문구는 강한 빨강보다 낮은 채도의 브라운, 오커, 다크 블루 계열을 우선 검토한다.
- 한 화면에서 강한 색은 1개, 보조 강조색은 많아도 1개까지만 허용한다.

### 7-4. 타이포그래피 방향

- 군더더기 없는 산세리프 사용
- 제목은 강하게, 본문은 조용하게
- 숫자와 단위는 읽기 쉬워야 한다

추천 역할:

- 페이지 제목: 24-28px
- 패널 제목: 14-16px
- 기본 본문: 13-14px
- 보조 설명: 12-13px

### 7-5. 간격 시스템

8px 기반 spacing 시스템을 사용한다.

예시:

- 패널 내부 기본 간격: 16px
- 블록 간 간격: 24px
- 컨트롤 라벨과 입력 사이: 8px
- 전체 좌우 패딩: 24px

### 7-6. 모서리와 선

- radius는 과장하지 않고 작게 유지한다
- 추천 radius: 8px, 12px, 16px 세 단계
- border는 1px 기준으로 일관되게 사용한다

## 8. 컴포넌트 설계 원칙

### 8-1. 버튼

- 우선순위는 `Primary`, `Secondary`, `Quiet` 세 단계면 충분하다
- `Upload`, `Download`만 1차 버튼으로 본다
- `Undo`, `Redo`, `Reset`은 2차 또는 조용한 버튼으로 둔다

#### 버튼 공통 규칙

- 최소 높이와 패딩이 있어야 한다
- 테두리 또는 명확한 배경이 있어야 한다
- hover 시 색 또는 border가 변해야 한다
- disabled는 흐리되, 버튼 형태는 유지해야 한다

#### `Primary`

- 가장 중요한 현재 행동
- 채워진 배경 + 높은 대비 텍스트
- 예: `Upload`, `Download`

#### `Secondary`

- 자주 쓰지만 1순위는 아닌 행동
- 채워진 옅은 배경 + 명확한 border
- 예: 효과 카드, 속성 패널의 세그먼트 버튼

#### `Quiet`

- 존재는 보여야 하지만 시각적으로 앞에 나오면 안 되는 행동
- 최소한의 배경 또는 투명 배경 + border
- 예: `Reset`

#### 금지 규칙

- 버튼이 텍스트 링크처럼 보이는 상태
- 정보 카드와 버튼이 동일한 시각 스타일을 갖는 상태
- hover가 거의 느껴지지 않는 상태

### 8-2. 세그먼트 컨트롤

- `Brush / Rectangle`
- `Blur / Pixelate`
- `Smooth / Solid Fill`

세그먼트는 현재 상태를 한눈에 보여주는 데 적합하므로 적극 사용한다.

추가 규칙:

- 비활성, 일반, 활성 상태가 즉시 구분되어야 한다
- 활성 상태는 색상뿐 아니라 배경 면적과 border 변화로도 보여야 한다
- 세그먼트는 정보 뱃지보다 훨씬 버튼답게 보여야 한다

### 8-3. 슬라이더

- 블러 강도, 브러시 크기, 블록 크기처럼 연속 값을 다룰 때 사용한다
- 슬라이더 옆에는 항상 현재 값 숫자를 보여준다

### 8-4. 상태 뱃지

- 파일 해상도
- action 개수
- 선택된 효과

단, 뱃지는 정보 전달용일 뿐 장식처럼 많아지면 안 된다.

구분 규칙:

- 뱃지는 버튼보다 작고 약해야 한다
- 뱃지는 눌릴 수 없어 보이는 것이 맞다
- 따라서 버튼과 동일한 높이, 동일한 대비, 동일한 hover를 사용하면 안 된다

### 8-5. 캔버스

- 배경과 구분되는 명확한 프레임을 둔다
- 그림자는 약하게만 쓴다
- 캔버스는 떠 있는 카드보다 작업 평면처럼 보여야 한다

## 9. 컴포넌트 구분 규칙

### 9-1. 레벨 1: 구조 컴포넌트

- 헤더
- 좌측 툴 레일
- 중앙 캔버스 프레임
- 우측 속성 패널

이 레벨은 화면 구역을 나누는 큰 덩어리다.

시각 규칙:

- 분명한 외곽
- 안정적인 Surface 톤
- 과한 강조 금지

### 9-2. 레벨 2: 인터랙션 컴포넌트

- 버튼
- 세그먼트
- 슬라이더
- 컬러 입력

이 레벨은 사용자가 실제로 조작하는 요소다.

시각 규칙:

- 구조 컴포넌트보다 더 분명한 경계
- hover / active / disabled 상태 제공
- 카드보다 눌릴 수 있어 보여야 함

### 9-3. 레벨 3: 정보 컴포넌트

- 상태 뱃지
- 메타 정보
- 안내 문구
- 설명 텍스트

이 레벨은 읽기 위한 요소다.

시각 규칙:

- 조용해야 함
- 눌릴 것처럼 보이면 안 됨
- 레벨 2보다 대비가 약해야 함

### 9-4. 레벨 4: 작업 대상

- 실제 이미지
- 캔버스 위의 작업 결과

이 레벨은 도구보다 항상 우선시되어야 한다.

따라서 도구 UI가 이 레벨을 압도하면 실패다.

## 10. Empty / Loading / Disabled 상태

### 9-1. Empty State

- 첫 진입에서는 업로드 CTA가 가장 먼저 보여야 한다
- 설명은 짧고 구체적이어야 한다
- 예: `이미지를 올리면 바로 블러 또는 가림 편집을 시작할 수 있습니다`

### 9-2. Loading State

- 이미지 로드 중에는 버튼 텍스트가 바뀌고 상태 텍스트가 함께 바뀌어야 한다
- 과장된 스피너보다 텍스트 중심 피드백이 낫다

### 9-3. Disabled State

- 비활성 컨트롤은 존재는 보이되, 현재 왜 쓸 수 없는지 이해 가능해야 한다
- 예: 이미지 업로드 전에는 다운로드와 도구 버튼이 비활성

## 11. 안티패턴

이번 앱에서 피해야 할 UI 방향은 아래와 같다.

- 미려해 보이기 위한 과도한 그라디언트와 글로우
- 기능보다 브랜딩이 앞서는 대형 헤더
- 패널마다 다른 radius, 그림자, 여백 사용
- 불필요하게 많은 색상
- 버튼과 정보 배지가 모두 강하게 보이는 경쟁적 화면
- 편집 기능보다 장식적 카드 레이아웃이 눈에 띄는 구조
- 버튼과 카드가 같은 모습이라 클릭 가능성이 즉시 읽히지 않는 구조
- 모든 패널이 같은 톤이라 정보 계층이 납작해 보이는 구조

## 12. 구현 우선순위

### 11-1. 1차

- 레이아웃 구역 정리
- 좌측 툴 / 우측 속성 패널 구조 확정
- 버튼 우선순위 정리
- 색상과 간격 시스템 정리

### 11-2. 2차

## 14. 현재 결론

이번 앱의 UI는 "미니멀한 편집기"가 아니라 "질서정연하고 조용한 편집기"를 목표로 한다.

핵심은 다음과 같다.

- 이미지를 중심에 둔다
- 도구와 속성을 분리한다
- 정보 위계를 분명하게 한다
- 색과 장식을 줄이고 구조와 비례로 품질을 만든다
- Rams의 원칙을 시각 장식이 아니라 시스템 설계로 번역한다
- 버튼은 버튼처럼, 정보는 정보처럼, 패널은 패널처럼 즉시 읽히게 만든다
